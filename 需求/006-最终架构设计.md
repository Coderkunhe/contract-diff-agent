# 最终架构设计

## 版本: v1.0-target
## 时间: 2026-05-23

## 七步流水线

```
① 文件→结构化JSON (算法)
       │  clauses tree + tables + attachments
       ▼
② 差异位置定位 (算法)
       │  条款树匹配, 找出增/删/改的节点
       ▼
③ 差异内容识别 Agent (LLM)
       │  对已对齐的条款做语义级差异解读
       ├── 不合格 → 循环(最多3次/条, 5次/章) ──┘
       ▼
④ 校验 Agent (算法L2 + LLM L3)
       │  L2: snippet原文存在性验证
       │  L3: 语义验证 (CONFIRMED/REJECTED/UNCERTAIN)
       ├── 不合格 → 携带驳回原因回③重试 ──┘
       ▼
⑤ 友好提示生成 (LLM)
       │  通俗语言、风险归类、关注方建议
       ▼
⑥ 人工逐条确认
       │  不确定项重点标注
       ▼
⑦ 输出存档 (JSON)
```

## 循环退出机制

| 条件 | 动作 |
|---|---|
| 单条变化重试 >= 3 次 | 标记 uncertain, 留人工确认 |
| 单章节总循环 >= 5 次 | 该章跳过, 记录到 unmatched_content |
| 整体 confidence < 60% 的条数 > 1/3 | 标记整体可信度 low |

## 分层职责

| 层 | 技术 | 职责 | 确定性 |
|---|---|---|---|
| ① 结构解析 | pdfplumber + 正则 | 提取条款树、表格、附件 | 确定性 |
| ② 差异定位 | 条款树匹配算法 | 找出对齐节点和增删改 | 确定性 |
| ③ 内容识别 | LLM | 语义差异解读 | 非确定性 → 需验证 |
| ④ 校验 | 字符串匹配 + LLM | 反幻觉、交叉验证 | 确定性兜底 |
| ⑤ 友好输出 | LLM | 通俗改写 + 风险归类 | 非确定性 |
| ⑥ 人工确认 | 人 | 最终裁决 | 确定性 |
| ⑦ 存档 | Python | JSON 输出 | 确定性 |

```json
    ]

  "tables": [{ "page": 26, "caption": "...", "data": [...], "full_text": "..." }],
  "attachments": [{ "title": "附件一", "full_text": "..." }]
}
```
