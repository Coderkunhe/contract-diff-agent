## 三大交付物

### 1. 自进化 V2 — 人工纠偏反馈闭环

**问题**：V1 只能从管线运行结果中学习风险模式，用户无法对 LLM 判断做纠正。Web UI 上的“确认”是 localStorage 本地存储，不持久化。

**方案**：完整的 3 态判决闭环：

```
Web UI (确认/驳回/纠偏) → PUT /api/results/{id}/verdicts
  → result JSON (human_verdict + original snapshot)
  → extract_learning() 提取纠偏模式
  → format_learnings_context() 注入下次 LLM 提示词
```

**关键设计决策**：
- `human_verdict.original` 快照原始 LLM 输出 — 学的是“LLM 错在哪”，不是“人改成了啥”
- 纠偏时覆盖 `risk_level`/`risk_categories`/`risk_note` — 用户修正即时生效
- 幂等：重复提交不更新时间戳，保留首次纠偏时刻
- `load_past_learnings(full=True)` 加载完整学习记录，含纠偏案例

**涉及文件**：
- `src/pipeline/learning.py` — extract/load/format/update 全链路
- `web/app.py` — verdict 端点 + results_page 数据透传
- `web/templates/results.html.jinja2` — 3 态按钮 + 纠偏内联表单
- `web/templates/learnings.html.jinja2` — 人工反馈趋势
- `specs/自进化框架.md` — 完整架构文档

### 2. “全是低风险” Bug 修复

**根因**：LLM 的 JSON 响应被 markdown code fence 包裹（```json ... ```），`json_repair` 无法解析，返回 `{}`。空 dict 导致 `enh.get("risk_level", "low")` 一律走默认值 `"low"`。

**修复**：
- `src/pipeline/classifier.py` + `src/pipeline/identifier.py`：新增 `_clean_llm_json()`，正则剥离 markdown fence 后再调 `repair_json`
- `src/pipeline/traditional_diff.py`：硬编码 `"low"` 改为启发式（removed→high, added→medium, modified→medium），附说明 `risk_note`

### 3. 行业语料注入

**问题**：LLM 提示词是通用的，对珠宝/法律行业合同的特定风险模式识别不足。

**方案**：分类器和增强器提示词各加领域示例：
- 分类器：8 条珠宝/法律合同 few-shot 示例（钻石净度、GIA 证书、贵金属印记、定制验收、寄售结算、运输保险、单方调价、管辖权）
- 增强器：10 种行业典型场景速查表

## 已知局限

1. **纠偏数据需手动预热** — `/learnings` 首次使用为空，需提前跑 2-3 轮比对
2. **离线模式风险等级粗糙** — 只有 high/medium 两档，无分类细度
3. **多模型池未在演示中展示** — 模型切换是静默的，观众看不到
4. **SSE 进度在 CLI 不可见** — 只在 Web 端有实时流

## 架构决策记录

1. **`human_verdict.original` 快照策略**：存原始 LLM 输出而非当前值，防止连锁纠偏混淆学习信号
2. **`load_past_learnings(full=True)` 不替换旧接口**：保持向后兼容，旧调用方不受影响
3. **`_clean_llm_json` 位置**：在 `repair_json` 之前执行，不依赖 JSON 结构
4. **传统算法优先不变**：LLM 增强仍是可选层，离线模式功能完整
