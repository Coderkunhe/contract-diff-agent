## 通俗流程（说人话版）

把上面的技术架构翻译成日常场景，总共四步：

### 第一步：你上传合同，系统出结果

两份合同扔进去 → 系统找出所有差异 → 每条差异标上风险等级和分类。这一步跟之前一样。

### 第二步：你看结果时，告诉它“对不对”

每一条差异下面有三个按钮：

> **确认** — “AI 说得对，没问题”  
> **驳回** — “AI 瞎说的，这条根本不是风险”  
> **纠偏** — “风险确实存在，但 AI 判错了——应该是高风险不是中风险”

你点点按钮，意见就保存到服务器上了。关掉浏览器再打开，还在。

### 第三步：系统从你的纠正里“长记性”

每次比对结束后，系统会自动翻看你对哪些判断做了纠正，把这些记录下来：

- “AI 把 X 判为中风险，人给改成了高风险” → 记一笔
- “AI 说这个变化跟交付时效有关，人改成结算风险了” → 记一笔
- “AI 说这是风险，人给驳回了” → 记一笔

### 第四步：下次再跑，系统提前“打预防针”

下一次你再上传合同做比对时，AI 看到的提示词前面会自动多出一段话：

> “注意：过往 5 次比对中，R03 付款与结算被你纠正了 3 次——AI 在这个分类上容易判错，请特别谨慎。”

相当于每次你纠正它，都在教它——**用得越多，它判断越准**。

---

### 一句话总结

> 不是“AI 输出什么就是什么”——**你每次纠正它，它都会记住，下次少犯同样的错。** 就像带新人：指出来一次错，下次他会注意。系统也是这个道理，不过它不会忘。

## 数据模型

### human_verdict（per-change，存储在 Result JSON 中）

```json
{
  "human_verdict": {
    "action": "confirmed|rejected|corrected",
    "corrected_risk_level": "low",
    "corrected_risk_categories": ["R01"],
    "corrected_risk_note": "纠正后的风险说明",
    "timestamp": "2026-05-24T10:30:00Z",
    "original": {
      "risk_level": "high",
      "risk_categories": ["R04"],
      "risk_note": "原始LLM风险提示"
    }
  }
}
```

三种 action：
- **confirmed**：人工确认，LLM 判断正确
- **rejected**：人工驳回，此变化不是真实风险（LLM 幻觉）
- **corrected**：人工纠偏，变化真实存在但风险评估有误 → 覆盖 risk_level/categories/note

`original` 快照 LLM 原始输出，是学习"LLM 错在哪里"的关键数据。

### Learning Record（per-run，存储为 `data/learnings/run-{job_id}.json`）

V2 新增字段：

```json
{
  "human_feedback": {
    "feedback_count": 12,
    "feedback_rate": 0.6,
    "confirmed_count": 8,
    "rejected_count": 2,
    "corrected_count": 2,
    "llm_error_rate": 0.333,
    "most_corrected_categories": [
      {"id": "R01", "name": "交付时效", "count": 3}
    ],
    "correction_direction": [
      {"from": "high", "to": "low", "count": 2}
    ]
  },
  "correction_examples": [
    {
      "change_id": "diff-003",
      "brief": "验收时间从30天调整为45天",
      "original_risk": "high",
      "corrected_risk": "low",
      "original_categories": ["R01"],
      "corrected_categories": ["R01"]
    }
  ]
}
```

### Index（`data/learnings/index.json`）

V2 新增 run_entry 字段：`human_confirmed`, `human_corrected`, `human_rejected`
V2 新增 global_trends 字段：`total_human_feedback`, `avg_feedback_rate`, `avg_llm_error_rate`

## Prompt 注入机制

每次 LLM 调用前，`build_classifier_prompt()` 和 `build_enhance_system_prompt()` 自动执行：

1. `load_past_learnings(limit=5, full=True)` — 加载最近 5 次完整学习记录
2. `format_learnings_context()` — 生成两部分内容：
   - **历史比对经验**：风险分布、高频类别
   - **人工纠偏经验**（V2）：LLM 常错类别 + 具体纠偏案例
3. 拼接在 System Prompt 最前面

注入是**非致命的**——任何磁盘错误或缺失文件都不影响主流程，静默回退到原始 Prompt。

## Web UI

### 结果页面（`/results/{job_id}`）

每个 change 卡片底部有三个按钮：
- **确认**（绿）：我同意 LLM 的判断
- **驳回**（红）：LLM 无中生有，这不是风险
- **纠偏**（紫）：风险评估有误 → 弹出内联表单，可修改风险等级/类别/提示

所有 verdict 通过 `PUT /api/results/{job_id}/verdicts` 持久化到服务器。页面刷新后从服务器加载已有 verdict。

底部状态栏显示：已审核 X/Y 条，含确认/驳回/纠偏分解。

### 进化记录页面（`/learnings`）

V2 新增：累计人工反馈数、LLM 平均误判率趋势。每个 run 卡片显示该次运行的确认/驳回/纠偏数。

## 扩展方向（V3+）

- **主动学习**：识别 LLM 高不确定性区间的变化，主动提示人工审核
- **纠偏回放**：将历史纠偏作为 few-shot examples 注入 Prompt（而非仅摘要）
- **模型选择优化**：基于 per-model 的误判率统计，自动选择最优模型
- **跨合同模式识别**：发现同一商户/行业的系统性风险模式
