## 通俗流程（说人话版）

把上面的技术架构翻译成日常场景，总共四步：

### 第一步：你上传合同，系统出结果

两份合同扔进去 → 系统找出所有差异 → 每条差异标上风险等级和分类。这一步跟之前一样。

### 第二步：你看结果时，告诉它“对不对”

每一条差异下面有三个按钮：

> **确认** — “AI 说得对，没问题”  
> **驳回** — “AI 瞎说的，这条根本不是风险”  
> **纠偏** — “风险确实存在，但 AI 判错了——应该是高风险不是中风险”

你点点按钮，意见就保存到服务器上了。关掉浏览器再打开，还在。

### 第三步：系统从你的纠正里“长记性”

每次比对结束后，系统会自动翻看你对哪些判断做了纠正，把这些记录下来：

- “AI 把 X 判为中风险，人给改成了高风险” → 记一笔
- “AI 说这个变化跟交付时效有关，人改成结算风险了” → 记一笔
- “AI 说这是风险，人给驳回了” → 记一笔

### 第四步：下次再跑，系统提前“打预防针”

下一次你再上传合同做比对时，AI 看到的提示词前面会自动多出一段话：

> “注意：过往 5 次比对中，R03 付款与结算被你纠正了 3 次——AI 在这个分类上容易判错，请特别谨慎。”

相当于每次你纠正它，都在教它——**用得越多，它判断越准**。

---

### 一句话总结

> 不是“AI 输出什么就是什么”——**你每次纠正它，它都会记住，下次少犯同样的错。** 就像带新人：指出来一次错，下次他会注意。系统也是这个道理，不过它不会忘。

## 数据模型

### human_verdict（per-change，存储在 Result JSON 中）

```json
{
  "human_verdict": {
    "action": "confirmed|rejected|corrected",
    "corrected_risk_level": "low",
    "corrected_risk_categories": ["R01"],
    "corrected_risk_note": "纠正后的风险说明",
    "timestamp": "2026-05-24T10:30:00Z",
    "original": {
      "risk_level": "high",
      "risk_categories": ["R04"],
      "risk_note": "原始LLM风险提示"
    }
  }
}
```

三种操作含义：
- **confirmed（确认）**：人工核实，AI 判断正确
- **rejected（驳回）**：人工判断这不是真实风险——AI 虚报了
- **corrected（纠偏）**：风险真实存在但 AI 判错了等级/类别 → 系统用人工纠正后的值覆盖原值

`original` 字段存的是 AI 的原始结论——系统要知道"AI 原来是怎么判错的"，而不只是"人把它改成了什么"。这是学习信号。

### 学习记录（Learning Record，每次比对生成一份，存为 `data/learnings/run-{任务ID}.json`）

V2 新增字段（通俗解读在括号里）：

```json
{
  "human_feedback": {
    "feedback_count": 12,          // 本轮你点了多少次按钮
    "feedback_rate": 0.6,          // 有反馈的差异占总差异比例
    "confirmed_count": 8,          // 点了多少次"确认"
    "rejected_count": 2,           // 点了多少次"驳回"
    "corrected_count": 2,          // 点了多少次"纠偏"
    "llm_error_rate": 0.333,       // 本轮 AI 的误判率 = 被驳回和纠偏的 / 总反馈
    "most_corrected_categories": [ // 哪些风险类别被你纠正最多次
      {"id": "R01", "name": "交付时效", "count": 3}
    ],
    "correction_direction": [      // 纠偏方向：从什么改成什么
      {"from": "high", "to": "low", "count": 2}
    ]
  },
  "correction_examples": [         // 具体纠偏案例（最多存 10 条）
    {
      "change_id": "diff-003",
      "brief": "验收时间从30天调整为45天",
      "original_risk": "high",     // AI 原始判断
      "corrected_risk": "low",     // 人改成什么
      "original_categories": ["R01"],
      "corrected_categories": ["R01"]
    }
  ]
}
```

### 汇总索引（Index，`data/learnings/index.json`）

V2 新增：每条运行记录带确认/纠偏/驳回计数；全局趋势增加累计人工反馈数、平均反馈率、AI 平均误判率。这个文件供进化记录页面和 AI 指令生成读取。

## 怎么把经验喂给 AI（原"Prompt 注入"）

通俗讲：每次调用 AI 之前，系统自动翻出最近 5 次你纠正它的记录，写成一段"嘱咐"，放在指令最前面。

### 具体流程

1. **翻阅历史** — 加载最近 5 次比对的完整学习记录（含你每次的确认、驳回、纠偏）
2. **生成两份"备忘录"**：
   - **历史比对经验**："之前几次比对中，风险集中在交付时效和结算条款"
   - **人工纠偏经验**（V2 新增）："你在 R03 付款与结算上纠正了 AI 3 次——这个分类它容易判错，这次请重点关注"
3. **拼接在 AI 指令最前面** — 相当于每次跟 AI 对话前，先提醒它"上次你在这类问题上出过错，注意点"

### 关键设计

- **打断网也不影响使用** — 读历史记录失败时，系统静默跳过，AI 照常工作，只是少了"嘱咐"
- **纠偏过的问题优先提醒** — 不只告诉 AI "有哪些风险类别"，而是告诉它"哪些你之前判错过"
- **只记纠偏的，不记确认的** — 纠正过的才需要提醒 AI 注意，确认过的说明它本来就会

## Web UI

### 结果页面（`/results/{job_id}`）

每条差异卡片底部有三个按钮：
- **确认**（绿）：AI 说得对
- **驳回**（红）：AI 在瞎说，这不是风险
- **纠偏**（紫）：风险真实但 AI 判错了 → 弹出表单，可改正风险等级/类别/说明

所有操作自动保存到服务器。刷新页面、换电脑打开，意见都在。

底部状态栏显示：已审核 X/Y 条，含确认/驳回/纠偏分解。

### 进化记录页面（`/learnings`）

V2 新增：累计人工反馈数、AI 平均误判率趋势。每次运行的卡片显示该轮的确认/驳回/纠偏数。

## 扩展方向（V3+）

- **主动提醒审核**：AI 在拿不准的变化上主动标注"建议人工复核"
- **纠偏案例回放**：把历史纠偏的具体案例喂给 AI 当参考（目前只喂了统计数据）
- **自动选最优模型**：统计每个 AI 模型的误判率，自动用最准的那个
- **跨合同找规律**：发现同一商户/行业中反复出现的风险模式
