### 技术踩坑记录

| 坑 | 根因 | 解决方案 | 状态 |
|---|---|---|---|
| GMI 模型名格式 | 版本号用点号非横线 (4.6 vs 4-6) | 修正为 anthropic/claude-sonnet-4.6 | ✅ |
| Tool calling 超限 | GMI 不支持大请求+tool schema | 改直接 JSON 输出 | ✅ |
| max_tokens 上限 | GMI ~3000 token 硬限制 | 调整为 3000 | ✅ |
| JSON 截断 | 3000 token 写不完所有变化 | json_repair 库自动修复 | ✅ |
| JSON 非法引号 | 模型中文字段含 ASCII 引号 | json_repair + 提示词约束 | ✅ |
| Streaming usage | Stream 对象无顶层 usage | 从最后 chunk 提取 | ✅ |

## 已知局限

### 🔴 核心瓶颈: max_tokens=3000
- 只能输出 ~9 条完整变化（diff_summary 报告有 25 条）
- 后 16 条因截断丢失
- 风险分类和未对齐内容部分被截断

### 解决方向（v0.3）
1. **分章节比对**: 按合同章节拆成多次 LLM 调用，每次 3000 tokens 只处理 2-3 章
2. **两阶段**: 第一阶段列出所有差异摘要（~500 tokens），第二阶段逐条详细分析
3. **换模型/渠道**: 尝试 GMI 上的 claude-opus-4.7 是否有更高限制
