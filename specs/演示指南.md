# 演示指南

> v1.5.0 — 3 分钟紧凑演示脚本，覆盖全部 7 大技术亮点。

## 系统亮点

| # | 亮点 | 一句话 |
|---|------|--------|
| ① | 条款树解析 | PDF/DOCX 提取 L1/L2/L3 层次结构 + 表格 + 附件 |
| ② | 条款对齐 | 章节编号偏移、中英文标题混排自动匹配 |
| ③ | 传统算法优先 | **零 LLM 依赖**，离线可用，SequenceMatcher 干重活 |
| ④ | LLM 增强 | 多模型并行，自然语言润色 + 风险分类，失败自动降级 |
| ⑤ | L2+L3 双重校验 | L2 字符串原文验证 + L3 独立 LLM 交叉校验，反幻觉 |
| ⑥ | 严谨模式 | 风险分类阶段独立盲审，不共享上下文 |
| **⑦** | **自进化 V2 人工纠偏闭环** | 确认/驳回/纠偏 → 服务端持久化 → 下次注入为"人工纠偏经验" |

### 第二幕：Web 上传 → SSE 实时进度 (0:15-0:45)

**亮点**：③ 传统算法优先 + SSE 流式推送

**操作**：
1. 打开 `http://localhost:8000`
2. 拖入两份合同 PDF（`docs/天猫服务协议2015(2).pdf` + `docs/天猫服务协议2026(2).pdf`）
3. 展示全页 overlay 拖拽动画
4. 点「开始智能比对」
5. SVG 环形进度条分阶段动画：① 解析 → ② 对齐 → ③ Diff → ④ LLM增强 → ⑤ 校验 → ⑥ 分类
6. 差异逐条实时弹出，右上角计数：已耗时 / 变化数 / 高风险数

**台词**：
> "进度是实时推送的，不是轮询。注意看——传统算法在第三步，不调任何 AI，已经找到了 393 条差异。后面 LLM 做的事：把每一条翻译成大白话，打上风险标签。"

**备选方案**（网络出问题时）：
> 直接跳到第五幕 CLI 离线演示，说："即使 Web 出问题——断网了系统照样跑。"

---

### 第三幕：结果页 (0:45-1:15)

**亮点**：③④⑤ 算法+AI+校验 三层组合

**操作**：
1. 左侧风险筛选栏：点「高风险」→ 列表只显示高风险项
2. 点风险类别 R01（交付时效）→ 筛选该类别
3. 右侧悬浮导航点阵：每个点代表一条差异，颜色区分风险等级
4. 点导航点 → 平滑滚动到对应卡片
5. 点开一条差异卡片，指给观众看：
   - `brief`：通俗摘要（"验收期限从 7 个工作日缩短为 3 个工作日"）
   - `risk_categories`：风险分类标签
   - `risk_level`：高/中/低
   - `attention_for`：关注部门（"采购部、法务部"）
   - `校验状态`：L2 原文验证通过 ✓

**台词**：
> "每条差异三件事说清楚：改了什么、有什么风险、该谁关注。注意底部的校验标记——L2 会把 AI 摘要里的关键词拿回原文做字符串匹配。AI 说的不等于全信，这是双重校验。"

---

### 第四幕：纠偏三态操作 (1:15-1:50) ✨ **V2 核心**

**亮点**：⑦ 自进化 V2 人工纠偏闭环

**操作**：
1. 找到一条 LLM 判「中风险」但实际应为「高风险」的差异
2. 点「纠偏」按钮 → 内联表单弹出
3. 改风险等级：中风险 → **高风险**
4. 选风险类别：勾选 R08（结算风险）
5. 填纠偏备注："验收时间缩短直接影响结算周期"
6. 点「提交纠偏」→ toast 提示"纠偏已保存"
7. 再点另一条 → 点「确认」→ 标记为"人工已确认"
8. **刷新页面** → 两条标记都在，证明服务端持久化
9. 底部统计栏显示：✓确认 N / ✗驳回 N / ✏纠偏 N

**台词**：
> "AI 不是百分之百对的。但你每一次纠偏，系统都会记住——数据写到服务端，不是浏览器缓存。下次比对时，你的纠偏会自动注入到 AI 提示词里：'R08 在过往 5 次中被人工纠正 3 次，请优先复核此类判断'。**人把关一次，系统记住一次——这叫闭环。**"

---

### 第五幕：进化记录 (1:50-2:15)

**亮点**：⑦ 自进化数据闭环

**操作**：
1. 点 header 「进化记录」→ 进入 `/learnings`
2. 展示累计统计卡片：总运行次数、平均差异数、总反馈数
3. 重点展示新加的「人工反馈趋势」区块：
   - LLM 错误率：X%
   - 最高频纠偏分类：R08 结算风险 (3 次)
   - 纠偏方向：high→low (2次), medium→high (1次)
4. 下方运行记录卡片：每次比对的 confirmed / rejected / corrected 数

**台词**：
> "数据说话。AI 在哪些分类上最容易被纠正，一目了然。下次比对时，这些数据自动回注到提示词里——**运行越多越准确，不是口号，是数据闭环。**"

---

### 第六幕：CLI 离线模式 (2:15-2:40)

**亮点**：③ Offline-First 架构

**操作**：
1. 切到终端，字号调大
2. 运行：
   ```bash
   make run-offline
   ```
3. 输出展示：
   - V1: 20 L1 章节, 75 L2 子条款
   - V2: 16 L1 章节, 40 L2 子条款
   - 匹配: 39 对, 新增: 2, 删除: 6
   - 传统算法识别到 **393 条差异**
   - 每条带启发式风险等级（删除→高风险，新增→中风险）

**台词**：
> "**没调任何 API**。传统算法是基础层，LLM 是可选的增强层。API 挂了、预算没了、网络断了——系统照样产出 393 条结构化差异。离线模式的每一条也有风险等级，虽然不如 AI 精准，但方向是对的。"

**台词**：

> "GitHub 开源，v1.5.0，122 个测试全绿。"

## 环境准备

```bash

make web
# → http://localhost:8000

# 2. 配置 API Key（演示完整 LLM 管线）
cp .env.example .env
# 编辑 .env，填入 LLM_API_KEY

# 3. 测试文件
ls docs/天猫服务协议2015\(2\).pdf   # V1 旧版
ls docs/天猫服务协议2026\(2\).pdf   # V2 新版

# 4. 预热 /learnings 数据（如果之前没跑过）
```

## 风险预案

| 风险 | 预案 |
|------|------|
| LLM API 超时/不可用 | 自动降级为离线模式，差异数不变，启发式风险等级兜底 |
| Web 服务起不来 | 切 CLI 模式，终端演示更直观 |
| 浏览器网络慢 | 离线模式演示是本地纯算法，不受影响 |
| /learnings 无历史数据 | 提前跑 2-3 轮预热（见上方环境准备） |
| 纠偏演示找不到合适差异 | 提前在数据里准备 2-3 条"中风险但实际应为高风险"的差异 |
